01

为什么要做

通用 AI 编排与 Magento 专项编排之间的差距

🏗️

版本升级困境

Magento 2.4.6 → 2.4.8 升级涉及 100+ 自定义模块的兼容性分析、废弃 API 检测、数据库架构迁移。通用 AI 缺乏 Magento 版本矩阵和 UCT 工具的深度集成。

🔧

日常维护低效

DI 配置、Plugin vs Observer 决策、EAV 属性设计、声明式 Schema 编写——这些需要深厚的 Magento 架构知识，通用 AI 往往给出次优方案。

💳

支付接入风险

支付网关集成涉及 PCI-DSS 合规、CSP 配置、Vault 集成、多货币/税务处理。任何疏漏都可能导致严重的安全和合规问题。

🎨

Hyvä 学习曲线

Hyvä 主题采用 Alpine.js + Tailwind CSS 替代传统的 jQuery/Knockout.js/RequireJS，ViewModelRegistry 模式与 Luma 截然不同，需要专门的知识体系。

04

Hyvä 优先原则

技术栈约束与架构决策

01

Hyvä First

所有前端决策默认采用 Hyvä 技术栈。仅在客户明确要求或第三方模块完全不兼容时，才考虑 Luma 方案。

02

No RequireJS

Hyvä 完全移除了 RequireJS/knockout.js 依赖。所有交互使用原生 Alpine.js，样式使用 Tailwind CSS。

03

ViewModelRegistry

使用 $viewModels->require() 模式在 phtml 模板中获取数据，替代传统的 Block 模式，实现更好的关注点分离。

04

CompatModule

为不兼容的第三方模块开发 Hyvä 兼容层，使用 CompatModuleRegistry 自动模板覆盖机制。

05

Magewire

复杂交互 UI（如结账流程）使用 Magewire 组件，实现类似 Laravel Livewire 的响应式体验。

06

Tailwind Config

tailwind.config.js 集成 @hyva-themes/hyva-modules，支持模块级别的样式扩展和自定义配置。
